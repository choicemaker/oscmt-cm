Chapter IX. Input/output plugins

1 Introduction

Input and output of records, record pairs, and marked record pairs in different formats is supported through input/output plugins in ChoiceMaker. Plugins exist for XML, flat files, and several relational databases. The same plugins handle I/O in the ChoiceMaker server and in ModelMaker.

1.1 Record, record pair, and marked record pair sources and sinks

Input and output can occur in three formats:

  1. Record. Individual records are read or written. Blocking is an example of this. Based on a single query record, a blocker returns all records from a database that may denote the same physical entity as the query record.
  2. Record pair. Pairs of records are read or written.
  3. Marked record pair. Pairs of records along with a differ/hold/match decision for each pair are read or written. Human-marked training data is an example of this.

In ChoiceMaker, an input is called a source and an output is called a sink.

1.2 Generated translators

Most I/O plugins generate custom classes based on the ChoiceMaker schema. Hence, a model must be rebuilt in ModelMaker upon adding or removing a plugin from the configuration or after changing the schema.

1.3 Configuration in the schema

Configuration information for each plugin can be stored in three places in the ChoiceMaker schema (Figure 65):

  1. Global configuration information is stored under the globalElement element (lines 4-7), which itself is a child of the global element. A plugin with name x, where x could be xml, flatfile, etc, may use a xGlobal element as child of the globalExt element. The exact content of xGlobal depends upon the individual plugin. For example, the xmlGlobal has attributes that govern what to do if an XML source contains unknown attributes or elements.
  2. Per node type information is stored under the nodeTypeExt element (lines 11-14). A plugin with name x may use a xNodeType element as child of the nodeTypeExt element. The exact content of xNodeType depends upon the individual plugin. For example, the dbNodeType has an attribute to define the view from which the data is retrieved.
  3. Per field information is stored under the field element (lines 15-18). A plugin with name x may use a xField element as child of the field element. The exact content of xField depends upon the individual plugin. For example, the flatfileField has attributes to define the position and width of a field.
