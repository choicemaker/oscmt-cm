8 Evaluation order and exception handling

Validity predicates as well as preconditions and values of derived fields are guaranteed to be evaluated in the order of declaration of the fields and nested node types in a ChoiceMaker schema file. Hence, in Figure 32 it would be legal to reference valid(lastName) in the validity predicate of firstName, but not the other way round.

Evaluation in stacked node types is depth first. The evaluation order among sibling nodes is explicitly left undefined and may change in future versions of ChoiceMaker.

Schema expressions may use Java methods that have non-empty throw clauses since all schema expressions are wrapped in a try-catch block.1 All exceptions may be logged by configuring logging through the configuration file. Only errors, e.g., OutOfMemoryError, cause program termination.

1In a future version, each expression might be wrapped individually.
