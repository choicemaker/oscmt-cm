4.1.3 Report files

The XML plugin also supports the reading of XML reports from the ChoiceMaker server. These reports contain a set of queries. Each query contains a query record and a set of match records. The XML plugin turns each match record into a pair together with the corresponding query record. This is useful for analyzing the accuracy of the ChoiceMaker server on actual production matches.

4.2 Configuration

The XML plugin supports the ChoiceMaker schema elements xmlGlobal, xmlNodeType, and xmlField. xmlGlobal has two attributes:

Name

Semantics

Use

Default

unknownElement

Controls the behavior when an unknown element is encountered in a XML source. The possible values are:

ignore: the element is ignored

warn: a warning is generated in the log

exception: an exception is thrown

optional

ignore

unknownAttribute

Same as above but for unknown attribute

optional

ignore

xmlNodeType is not used currently.

xmlField has only one attribute use that controls whether the field exists or not (Section 1.3.2).

The XML plugin does not yet support multiple configurations.

4.3 ModelMaker GUI

Figure 67 shows the XML marked record pair source configuration dialog, which appears when a user wishes to modify or create an XML marked record pair source. To use an existing XML file, the user enters the name of the source in the Source name field and the name of the file into the Source file field. The Browse... button may be used to locate the file.

The location of the XML file can be stored either relative to the descriptor file or as an absolute path. If it is stored relative to the descriptor file, copying both the descriptor and the XML file to a different location works seamlessly. Using an absolute path, the descriptor file by itself can be copied around while the XML file stays in the same directory. A relative path is the default. Note that the choice is governed exclusively by the drop down selector. E.g., if Relative to Descriptor is selected, the location is stored as a relative path even if the user enters an absolute path, manually or through the file chooser.

The source descriptor file (see above) gets created by pressing the OK button. The Cancel button closes the dialog without enacting any changes.

The XML I/O plugin supports the generation of XML source data from other sources, through the toggle button Generate new source file / Use existing source file (Figure 68). All of the above parameters must be filled in and a model must be selected to use the source generation. From the Source list, one or more sources must be selected. Their contents are inserted in the list order into the new source(s).

The data can be distributed over multiple sources by choosing a round robin scheme and/or a maximum number of pairs per file. A separate source descriptor is created for each file. This can be useful when, for instance, one wants to divide a single file into two files, one for training and one for testing.

With round robin over n files, the first pair goes into the first file, the second pair into the second file, etc. The nth pair is placed in the nth file and the n+1st pair goes again into the first file.

If the maximum number of pairs is set to a positive number m, a new file is created once the first file contains m records.

Round robin and maximum size may be combined. If the round robin parameter is n and the maximum size m, the first pair goes into the first file, the second pair into the second file, etc. The nth pair is placed in the nth file and the n+1st pair goes again into the first file. After the m*nth pair, the n files are closed and a new set of n files with corresponding source descriptors are created. This process is repeated after every m*n pairs.

The GUI for XML record sources is analogous, except that it does not currently allow the generation of source data.
