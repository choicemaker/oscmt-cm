9 Holder classes and translators

This section deals with the internal representation of data and may be skipped on a first reading.

ChoiceMaker creates custom Java holder and translator classes based upon the schema. For each node type x, ChoiceMaker generates four holder classes and interfaces:

  • Interface XBase is the base interface of all generated classes. It contains methods getF() and isFValid() for each non-derived (JavaBeans source) field F. Furthermore it contains methods getY() and getY(int index) for each nested node type Y.
  • Class XHolder implements XBase and also contains setter methods. This class conforms to the JavaBeans specification. It is used to exchange data between ChoiceMaker Server and clients.
  • Interface X extends XBase with getter and is valid methods for derived fields. It can be used in methods called from ClueMaker and ChoiceMaker Schema expressions.
  • Class XImpl implements X. It is used internally to represent nodes in memory.

The first three types, XBase, XHolder, and X, are put into a package p.s, where p is the package declared in the clue set and s is the name of the schema. Class XImpl is put into package p.internal.s.

ChoiceMaker needs to exchange record data in a wide variety of formats, such as databases, XML, flat file, and CORBA. Each exchange requires a translation from or to the internal format described above (data binding). The plugins and customizations that provide support for a specific I/O format are responsible for creating the required translator classes (Chapter IX).

These classes are created when the model is built in ModelMaker (Chapter II.6). Custom generated holder and translators are considerably faster than generic trees and translators that use reflection. The gain in run-time performance greatly outweighs the cost of generation translators at compile time.
