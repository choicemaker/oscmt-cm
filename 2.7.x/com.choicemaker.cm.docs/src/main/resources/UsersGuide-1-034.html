12.1 Matcher

The matcher dialog (Figure 26), which can be displayed by choosing Tools->Match..., allows records from two sources to be matched against each other or records from a single source to be de-duplicated.

The Input sources section defines the record sources to be matched. The matcher takes as input record sources. Record sources are collections of records. Typically, they are stored in a flat file, XML file, or a database. Each record source has a descriptor. Record source descriptors are like marked record pair source descriptors (Section 7.1), but with the extension `.rs'. The Preview button allows the user to verify that the correct source is selected. The preview (Figure 27) is displayed in a separate dialog that uses the same layout as the human review panel (Section 10.2).

The Blocking section defines the blocker to be used. The ChoiceMaker core distribution only includes the All blocker, which returns all records from the small source. That is, all records from the large source will be paired up with all records from the small source and sent to scoring. This yields the lowest number of false negatives at the cost of a very slow execution.

The optional blocking plugin (Chapter IX.7) contains more sophisticated blockers.

Scoring is based on the active model and the present thresholds. Therefore, this dialog is only available if a model is open.

The Filter works like the filter described in Section 11. By default, record pairs with a differ decision are filtered out. Additionally, there is an option to exclude matches to the same id. The matches can be sorted:

  1. No sorting: The pairs appear in arbitrary order.
  2. Lexicographical: decision, probability decreasing: The pairs are order first by decision (match, hold, differ) and then by probability decreasing within each decision category.
  3. Probability decreasing: The pairs are ordered by match probability decreasing.

If desired, only the n first matches (after sorting) for each record from the large source are output. Records that are filtered out in this last step may still count for the limiter.

Finally, the Output section defines in which format and where the output should be stored. There are two supported output formats: a marked record pair source and a comma separated flat file containing the two ids, the decision and the probability. The output, that is all pairs accepted by the filter are stored in the specified location. Note that any source data in the specified sink are overwritten.

12.2 String comparison functions

The string comparison function utility (Figure 28) allows the exploration of the various approximate string matching functions (Chapter VI.1) bundled with ChoiceMaker. It is displayed by selecting Tools->String Comparison Functions....

12.3 Collections Lookup Utility

The collections lookup utility (Figure 29) allows the user to explore the various Sets, Maps, and Relations in the current model (Chapter VI.5). The user selects the appropriate Set, Map, and Relation from the respective drop-down menus. The contents of the selected Set and Map are displayed in the respective tables.

As the user enters a String in the Key field, the Sets and Maps relations tables scroll to the appropriate location. Any entries in the selected Set and Map that begin with the specified key are highlighted. If the key matches an entry in the specified Relation, its entries are displayed in the Forward and Backward Set tables.

12.4 Holder classes and database objects generator

The holder classes and database objects generator utility (Figure 30) allows the user to generate various objects for the configured production probability models. By default the output goes to <generatorCodeRoot>/out.

The following objects can be generated:

  • Production Models Jar File: The jar file models.jar containing the models and configuration file. This is used in ChoiceMaker Server J2EE. When generating models.jar, ModelMaker needs to contain all the generator plugins deployed in ChoiceMaker Server J2EE, or a method/class not found error will occur at runtime. Additional plugins do not present a problem.
  • Holder Classes Jar File: The jar file holderClasses.jar containing the JavaBeans holder classes (Chapter III.9). This is used in clients of ChoiceMaker Server J2EE not contained in ChoiceMaker.ear.
  • Holder Classes Javadoc (Zipped): JavaDoc of the holder classes.
Depending upon the installed plugins, additional options such as SQL Server Custom Objects (DDL) may be generated.
